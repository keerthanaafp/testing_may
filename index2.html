A / 01
New Mobile · Input

Product catalogue input

A new mobile element for letting users pick products and place an order.

You can now add a product catalogue to any mobile view. Users can browse products, pick more than one, set quantities, and see their cart update as they go.

What you get

  • Pick multiple products with quantities (the old version only allowed picking one).
  • Filter by category and review everything in a summary screen before submitting.
  • Barcode scanner with camera controls — torch, focus, and keep-scanning mode.
  • Required check — the form won't submit if no product is chosen.
  • Live cart always visible at the bottom so users know what they've picked.
Heads up The element used to be called catalogue. It is now product_catalogue. Old views that use the old name will need updating — they won't change on their own.
What to test Scan a barcode, filter by category, pick multiple products, change quantities, try to submit with an empty cart (should be blocked), and check that old views using catalogue get flagged.
B / 02
Mobile · Web Improvement

Chat: photos work, messages stay in order

Mobile users can now attach photos. The web chat scrolls and orders messages correctly.

Chat got fixes on both mobile and web this week.

On mobile

  • Take multiple photos in one go and send them all in a single message.
  • Better preview — see your photos and remove any of them before sending.
  • Photos are saved in a format other widgets can read, so you can show attached images elsewhere in the app.

On web

  • Chat now stays scrolled to the latest message — image-heavy chats no longer break the layout.
  • Messages show in the right order and duplicate messages no longer appear when the chat reloads.
What to test Mobile: take many photos, remove some, send the rest. Web: open a busy chat with images, check that it scrolls to the bottom, refresh, and confirm there are no duplicates or out-of-order messages.
C / 03
Mobile · Payments Security

Tap to Pay — safer setup

The Stripe secret key now lives on the server, not the phone.

Last week we launched Tap to Pay. This week we made it safer. The Stripe secret key used to be sent to the mobile app. Now the mobile app asks our server for a one-time token, and the server talks to Stripe instead.

For the user, nothing changes — payments work the same way. The change is invisible. But if someone tampered with a phone before, they could try to read the Stripe key. They can't anymore.

QA blocker This is a security-sensitive change. Test the full Tap to Pay flow end-to-end on a real device before release. A bug here blocks the release.
What to test Tap to Pay on a real device (start to finish), test the simulator mode too, and confirm the secret key never appears in network traffic from the phone.
D / 04
New Mobile · Portal

Offline mode

Field users can keep working without internet. Forms send out once they're back online.

The mobile app now has an offline mode. If a user loses internet in the field, they can still:

  • See data that was already loaded.
  • Fill in and save forms — the app holds them locally.
  • When the phone gets back online, the saved forms send automatically.

Offline mode is off by default. You have to turn it on for each app from the portal. Apps that aren't turned on behave exactly like before.

What to test Turn it on in the portal, put the phone in airplane mode, browse cached views, fill in a form, then come back online and check the form is sent. Also check that an app where it's turned off still needs internet.
E / 05
Web · Auth Improvement

Login lands you where you wanted to go

Plus: URL settings like ?foo=bar now reach the view automatically.

Two small but useful URL fixes:

What's new

  • Logging in keeps you on the right page. Open a page that needs login while logged out. You get sent to the login screen. After logging in, you land on the original page — not the dashboard.
  • URL settings reach the view. If your URL has ?foo=bar, the view can read foo straight from arguments. No setup needed.
  • API keys come from a registry now instead of being hardcoded. This sets us up to let customers manage their own keys soon.
What to test Open a protected URL in a private window, log in, and check you land on the original URL. Try a URL with ?key=value and confirm the view receives it. Try creating and rotating an API key.
F / 06
New Web · Navigation

Lists refresh after you edit something

No more stale lists. Edit on the detail page, hit back, and the list updates.

Before this week, you'd open a list, click into a row, change the status, hit back — and the list still showed the old status. You had to refresh the whole page. Annoying.

Now the list updates on its own when you come back from a detail page where you changed something. No refresh needed.

What to test From a list, click into a row, change something (status, name, delete), hit back. The list should show the change without you refreshing. Try it across same-app and different-app navigation.
G / 07
Sheets Improvement

Filters can now use AND and OR

Build more flexible filters by mixing both connectors.

Sheets filters used to only let you stack rules with AND — every rule had to be true. Now you can use AND or OR between rules.

That means you can ask for things like "status is open AND (priority is high OR assignee is me)" right from the filter UI.

What changed

  • Each new rule can join the previous one with AND or OR.
  • The filter chips show how the rules are connected so you can see what's going on.
  • Clearing one rule keeps the others in place.
What to test Try filters with only AND (should work like before), only OR, and mixed AND+OR. Test with date columns and dropdown columns. Clear one rule and check the others stay.
H / 08
New Web · Widgets

Two new widgets + an HTML widget fix

TieredOptions for pricing-style lists, ExpandableCard on web, and a big rebuild of HTML display.

New widgets

  • TieredOptions — a layout for picking one option from a few tiers. Good for plan pickers, package choices, or any "pick one of these" screen.
  • ExpandableCard — a card that opens and closes. The mobile app got it last week; the web version is here now so both platforms look the same.

HTML display widget — rebuilt

The HTML display widget had a problem: the HTML you put inside it would sometimes pick up styles from the rest of the page, and vice versa. Things looked broken.

It's now wrapped in an iframe — a sealed box for the HTML — so the embedded content stays separate from the host page. This fixes a class of layout bugs on both web and mobile.

Polish

  • Text widgets re-render less often, so screens feel snappier. You can also force a refresh when you actually need one.
What to test Any view that uses HTML display — check it on both web and mobile (this is the biggest change). Try TieredOptions selection. Confirm ExpandableCard works the same on web as on mobile.
I / 09
Web · Admin Improvement

Team members — search added

Type a name or email to find someone. No more scrolling through long lists.

The team members popup now has a search box. Type a name or email and the list filters as you type. Useful when your team has more than a handful of people.

What to test Search by name, search by email, check the empty state when nothing matches, and clear the search to get the full list back.

Backend Platform

Rate limits, India region & small fixes

A bunch of smaller backend and web improvements.

Backend

  • One noisy API key won't slow down the others. Rate limits now apply per key, not across all of them.
  • File uploads work with public-access tokens — public-access users can upload files now.
  • India region setup — database and server settings updated for the IN cluster.

Small web fixes

  • Signature input — the error message clears once you've drawn a signature.
  • Image upload — closing the upload dialog no longer accidentally moves you to another page.
  • Form validation — gutter and row validation correctly looks at the right parent.
  • Calendar header cleaned up.
  • Offline mode toggle shows up in the app config (for the new mobile feature).
  • Image visibility condition fix.

Mobile platform

  • iOS bumped to 1.0.15, Bluetooth permissions added.
  • macOS build fix.
  • Better in-app logging — see API calls and queries in staging.
  • ExpandableCard widget (mobile half of widget H).

Docs site

  • ⌘K / Ctrl+K command palette — quick search and jump around the docs.
Release gate The web app's environment was switched to staging mid-week for testing. Make sure the production build is NOT set to staging before release.
What to test Hit one API key hard — confirm you get 429s and that a second key isn't blocked. Confirm public-access tokens can upload files. Run smoke tests on the IN cluster. Confirm the production build isn't pinned to staging. Try ⌘K in the docs.

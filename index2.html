A / 01
New Mobile · Input

Product catalogue input element

A new mobile input that turns any view into a multi-select ordering surface — with categories, scanning, and live cart.

The mobile app gets a brand-new product_catalogue input — a full ordering surface you can drop into any view. Pick products, filter by category, scan barcodes, and watch the cart update live. This was previously just catalogue; it has been renamed and rebuilt around multi-select ordering as the primary flow.

What landed

  • Multi-select ordering with quantity per product, replacing the old single-pick behaviour.
  • Category filtering & summary screen with improved validation, so users see what's selected before submitting.
  • Dedicated scanner screen with camera controls — torch, focus, and continuous scanning.
  • Required validation on the catalogue input plus a live cart-state display so the form state is always visible.
  • Entry-point UI refactor and a card-component redesign paired with the matching web release.
  • Docs renamed — every reference from catalogue to product_catalogue.
Heads up Anything that referenced the old catalogue element type needs to be updated to product_catalogue. Existing views won't auto-migrate — flag any failing references during QA.
What to test Scan flow, category filter, multi-select with quantity, cart updates, required validation blocking submit when empty, and renamed product_catalogue references in existing views.
B / 02
Mobile · Web Improvement

Chat: multi-capture, attachments & ordering

Image attachments land properly on mobile; web fixes scroll-to-bottom and message ordering.

The chat widget had a focused week across both platforms. Mobile picks up real attachment support; web ships reliability fixes for the long-running scroll and ordering issues.

Mobile

  • Multi-capture camera screen — capture multiple photos in one flow and attach them as a single message.
  • Redesigned attachment UI for previewing and removing photos before sending.
  • Documented inputData format for attachments so views and queries can read attachment URLs reliably.

Web

  • Scroll-to-bottom reliability — the message list now actually stays pinned to the latest message; image-heavy threads no longer break the layout.
  • Message sort & dedup refactor — out-of-order arrivals are placed correctly and duplicate messages no longer appear when a thread reloads.
What to test Mobile: multi-capture, attachment preview/remove, send with multiple images, attachment URLs visible via inputData. Web: scroll-to-bottom after image load, message ordering after reconnect, dedup across rapid loads.
C / 03
Mobile · Payments Security

Tap to Pay — server-side token fetching

The Stripe Terminal flow now mints connection tokens on the backend. The mobile never sees the secret key.

Following last week's Tap to Pay launch, the connection-token path has been fully migrated server-side. The mobile app no longer holds or fetches the Stripe secret — it asks the backend for a token, and the backend resolves the secret reference and talks to Stripe.

Functionally nothing changes for end users; security posture changes meaningfully. A network inspector on a tampered device cannot now leak sk_*.

QA blocker This is a security-sensitive migration. Verify Tap to Pay still completes end-to-end on a prod-like config — both the simulator path and a real device. A regression here is a hard block on release.
What to test Real-device Tap to Pay end-to-end, simulator path, network inspection confirming no sk_* leaves the backend, and graceful handling when the secret reference is misconfigured.
D / 04
New Mobile · Portal

Offline mode

Mobile picks up offline support, with a portal toggle to opt apps in.

The mobile app now supports an offline mode for field users who lose connectivity mid-shift. Cached data continues to render and form submissions are queued locally, sync resumes when the device comes back online.

Offline mode is per-app and opt-in from the portal. Apps that aren't whitelisted behave exactly as before. The portal toggle and the mobile feature shipped together so they can be tested end-to-end.

What to test Toggle on in portal → app reads cached views offline → form submissions queue → device returns online → queued mutations flush. Also test that a non-whitelisted app still requires network as before.
E / 05
Web · Auth Improvement

Deep links & URL query args

Logged-out users now land where they were going. Query strings flow into view arguments.

Two related fixes to the way the web app handles URLs:

What's new

  • Deep links survive login. Hit a protected URL while logged out, get redirected to login, and after authenticating you land on the original URL — not the dashboard.
  • URL query parameters populate view arguments. Pass ?foo=bar on a view URL and {{arguments.foo}} resolves to bar inside that view — no extra config needed.
  • Registry-backed API key management — API keys are now driven by the integration registry instead of hardcoded paths, paving the way for tenant-managed keys.
What to test Open a protected URL in a private window, complete login, confirm you land on the original URL. Pass ?key=value on a view URL and confirm the matching argument is populated. Confirm API key registry list and create/rotate flows work end-to-end.
F / 06
New Web · Navigation

Back-navigation refresh queue

Mutations on a detail view now propagate back to the list automatically.

When you navigate from a list to a detail view, change something, and hit back, the parent list now re-fetches automatically. No more stale rows that require a manual page refresh to reflect the change you just made.

Under the hood, navigation maintains a small pending refresh queue — keys that were mutated downstream get queued for re-fetch on return. The queue is flushed on the back transition.

What to test Navigate list → detail → mutate (update a status, change a name, delete a row) → hit back. The list should reflect the change without a hard reload. Try across both same-app and cross-app navigation.
G / 07
Sheets Improvement

AND/OR connectors for multi-rule filters

Sheets filters now support mixed boolean logic, not just AND-stacks.

Building on last week's advanced filtering, Sheets now lets you compose filter rules with both AND and OR connectors. You can express predicates like "status is open AND (priority is high OR assignee is me)" directly in the filter UI.

What changed

  • Each filter rule can be joined to the previous one with AND or OR.
  • Mixed AND+OR groups evaluate left-to-right with standard precedence — verify on complex filters with QA.
  • The filter chip row reflects the active boolean structure, so users can see what's being combined and how.
What to test Combined filter conditions, especially mixed AND+OR, with date and multi-select column types. Confirm filter chips reflect the boolean structure and clearing one rule preserves the rest.
H / 08
New Web · Widgets

TieredOptions, ExpandableCard & the HTML iframe refactor

Two new widgets land. The HTML display widget gets rebuilt around iframes.

A meaningful widget release — two new widgets and a major refactor under the existing html_display element.

New widgets

  • TieredOptions — a pricing/plan selector with stacked tiers. Designed for plan-picker, package-selection, or any one-of-N comparison surface.
  • ExpandableCard on web — mirrors the new mobile ExpandableCard widget for consistent UX across platforms. Syntax and logic refined further this week.

HTML display — iframe refactor

The html_display widget has been refactored to render inside an iframe as a platform view with explicit DOM-mutation handling. This isolates embedded HTML from the host page's styles and scripts and fixes a class of layout-bleed bugs that were affecting both web and mobile.

Polish

  • Text reactivity optimization plus explicit data refresh — fewer unnecessary re-renders, but explicit refresh hooks where you need them.
What to test Any view embedding HTML content — verify rendering, scroll behaviour, and styling on both web and mobile (the iframe refactor is the highest-risk piece). Smoke-test TieredOptions selection flows. Confirm ExpandableCard parity between web and mobile.

Web · Admin Improvement

Team members dialog — search added

Finding the right teammate in a large org is now actually possible.

The team members dialog now has a search input. Type a name or email and the member list filters live. Useful in tenants with 50+ users where scrolling the full list is slow.

What to test Search by name and by email, empty-state when nothing matches, and the clear-search behaviour returning to the full list.
J / 10
Integrations Migration

QuickBooks → Nango migration

QB moves off our custom OAuth onto the Nango integration platform — and the old gateway goes away.

QuickBooks now flows through Nango for OAuth and webhook routing instead of the custom plumbing we shipped two weeks ago. The custom QuickBooks gateway and routing infrastructure have been removed across the stack.

What was touched

  • Main backend — Nango proxy added for OAuth + webhook routing; old QuickBooks gateway and routing infrastructure removed.
  • Node backend — QuickBooks migrated from custom OAuth to the Nango platform.
  • Web app — QuickBooks integration removed from the web app surface (it's now driven entirely via Nango).
  • Portal — hardcoded QuickBooks integration label removed. Connected integrations now render from the registry.
QA blocker The custom QB path is gone — any regression here is a hard block. Run the full end-to-end QuickBooks flow via Nango: connect → push customer → push invoice → pull payment.
What to test QuickBooks connect/disconnect, push customer, push invoice, pull payment confirmation via webhook — all via Nango. Confirm the Portal integration list renders from the registry rather than the removed hardcoded label.

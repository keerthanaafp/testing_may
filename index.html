New App Builder

Preview your changes on save

A preview option in the JSON view editor — click to see the change live, right after saving.

When editing a view's JSON in the app builder, you now get a preview option. After saving, simply click Preview to view the changes in their rendered state — no need to leave the editor, switch tabs, or reload the app to verify how the edit looks.

This shortens the edit-save-check loop dramatically, especially when iterating on layout, conditional visibility, or widget configuration.

What to test Preview button visibility after a save, accurate rendering of the saved JSON, and behaviour when previewing an unsaved or invalid JSON state.

New App Builder

Undo support in the JSON editor

Ctrl + Z now reverts JSON edits — even after a change has been applied.

Undo has been brought into the JSON view editor. Press Ctrl + Z (or ⌘ + Z on macOS) to step back to the previous JSON state — even after you've made a change to the view. No more manually re-typing or copying the old config back.

This works in tandem with the new preview option: tweak the JSON, hit preview, decide you don't like it, and Ctrl + Z brings you back to where you started.

What to test Single-step undo, multi-step undo through a chain of edits, behaviour after a save, and macOS shortcut (⌘ + Z) parity.

New Portal

Portal members, RBAC, audit log & password change

The internal Portal is no longer a single-user tool — full team management has landed.

Four related Portal features shipped together and should be treated as one system:

What's new

  • Members management — invite teammates into Portal, assign a role, remove them. Sessions invalidate immediately on removal.
  • Role-based access control (RBAC) — three tiers: owner, admin, read-only. Enforced in the UI and on the backend — a read-only member calling an admin endpoint directly will get rejected.
  • Audit log — every meaningful Portal action (member added, integration disconnected, AI flag toggled, etc.) is recorded with actor, target, timestamp, and a drill-in for details.
  • Change password — Portal members can now rotate their own password from the new modal; the backend validates the current password and enforces password strength rules.
Why it matters Until this week, everyone with Portal access had god-mode. From now on, customer-facing ops people can be granted read-only access without risk of accidentally force-disconnecting a tenant's integration.

New Security

Tenant-scoped secrets management

Stop pasting API keys into action configs. Reference them by name; we resolve them server-side.

Admins can now create encrypted secrets per tenant under Settings → Secrets and reference them inside any server-routed API action as {{secrets.NAME}}. Substitution happens on the backend just before the outbound HTTP request fires — the plaintext value never reaches the browser or the mobile device.

The rules

  • Write-only. Once a secret is saved, no client can read it back. You can rename, replace, or delete — but never view.
  • Admin-only. Non-admins get 403 on the settings page and on the API.
  • Server-routed only. Secrets resolve inside the /data/serverAPICall endpoint. A plain browser-side api action would send the literal {{secrets.NAME}} string upstream — so always route secret-bearing calls through the server proxy.
  • Missing secret = 400. If a referenced secret doesn't exist, the request is rejected with Unknown secret(s): NAME and is never dispatched.
Mobile The mobile app picked up matching docs for the same syntax — Stripe keys, signing secrets, etc., all live server-side and get substituted in transit.
What to test Secret creation, admin-only visibility, server-side substitution in api actions, and missing-secret error handling.
E / 05
Sheets Improvement

Filters, bulk operations & smarter CSV import

Sheets gets the heavy-data improvements you've been asking for.

Sheets has had a meaningful upgrade across filtering, selection, and import:

What's new

  • Advanced multi-rule filtering — add multiple filter rules that combine with AND across mixed column types (string, number, date, multi-select). Each operator is column-type aware.
  • Date filters now match the full day. Setting date == 2026-05-12 matches every row on that day rather than requiring an exact timestamp match.
  • Bulk actions on multi-select. Checkbox-select rows in any list widget, then run bulk delete, bulk update, or any custom action across all selected rows.
  • Pagination as a first-class widget — extracted from the data grid so you can choose page sizes (10 / 50 / 200) consistently, with horizontal scrollbar always visible on wide tables.
  • CSV import accepts blank IDs. If your CSV has no id column or leaves it empty, the backend now auto-generates IDs instead of rejecting the row.
  • Schema reconcile — from Portal you can compare a saved sheet config against the live DB schema and resolve any drift.
  • Admin-gated. Sheets is now hidden from non-admin users to prevent accidental schema edits.
Re-import caveat Re-importing the same CSV will currently not dedupe against existing rows — confirm with product whether idempotent re-imports are part of the intended behaviour.
What to test Multi-rule filters across column types, date day-range matching, bulk actions, pagination, blank-ID CSV import, and admin gating.

New Mobile · Payments

Stripe Tap to Pay on mobile

Real Stripe Terminal integration — secret key held on the backend, mobile never sees it.

Tap to Pay moves from a mock/device-side flow to real Stripe Terminal, with the Stripe secret key held on the backend. Field technicians can accept card payments directly through the mobile app — and the device never holds sk_*.

How it works

  • New backend endpoint POST /stripe/terminal/connection-token resolves {{secrets.NAME}} against fp_as_secrets, mints a Stripe Terminal connection token, and logs the secret-usage event on both success and failure.
  • Mobile no longer holds the Stripe secret. The Stripe service takes a fetchToken callback that calls the backend; the ViewController passes secret references ({{secrets.STRIPE_KEY}}) instead of resolved values.
  • Real Stripe connection_tokens API replaces the previous mock, with simulator mode via stripe_simulated for development without NFC hardware.
  • Configurable timeout via timeout_seconds; collection starts before the tap prompt so the reader is ready.
  • Both succeeded and requires_capture outcomes are treated as success.
  • Docs published — new tap_to_pay action reference, setup checklist, and error table in the docs site and in-app docs.
Security note The raw Stripe secret key now lives only in fp_as_secrets on the backend. A network inspector on the mobile device should only see the connection-token call to our backend — not a direct Stripe API call carrying sk_*.
What to test Simulator path with stripe_simulated: true, real device path verifying no sk_* ever leaves the backend, misconfigured secret-name error handling, requires_capture as success, timeout expiry, and secret-usage audit log entries on both success and failure.

New Charts

Horizontal bar chart variant

A new chart orientation for category-heavy data and long labels.

The bar chart widget now supports a horizontal orientation in addition to the existing vertical layout. Use it when you have many categories, long category names, or want to emphasize ranking — bars extending rightward read more naturally than rotated x-axis labels.

When to use it

  • Long category labels — labels sit flush on the y-axis with full breathing room, no rotation or truncation.
  • Many categories — vertical scrolling handles tall lists better than horizontal scrolling on narrow screens.
  • Ranking visualizations — top-N leaderboards, sorted comparisons, or any "biggest to smallest" display.
What to test Horizontal rendering with long labels, many categories, and on narrow viewports.

A set of long-standing rendering issues in the vertical bar chart were resolved this week. Previously, long category names would clip at the edge of the chart area or overlap adjacent labels — both now handled cleanly.

New Mobile Widget

Mobile HTML Display widget

An inline HTML renderer for the mobile app — auto-sizing, themable, JS-capable.

A new html_display widget that renders raw HTML inline on mobile. Use it for rich text, formatted reports, or any server-generated markup that needs to live inside a view.

What it does

  • Auto-sizes to content — measures the rendered height and resizes itself. Pass a fixed height if you want to override.
  • Accepts fragments or full documents — pass <p>Hello</p> and it inherits your chosen styling; pass a full <!DOCTYPE> document and it loads as-is.
  • Themable — control background_color, text_color, link_color, font_size, font_family, and content_padding, or inject custom rules via css.
  • Template-aware — supports {{}} and <!...!> syntax inside the HTML, so values bind dynamically.
  • JS + zoom toggles — enable_js (default on) and enable_zoom (default off).
  • Conditionally visible via visibility_condition, like other widgets.
What to test Fragment and full-document rendering, auto-sizing vs fixed height, theming fields, template substitution, and visibility conditions.
